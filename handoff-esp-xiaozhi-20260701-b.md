## 语音回复整改（后端，`myagent` commit `6625a8c`）

用户反馈链：TTS 整段处理完才播(首字等极长) → 语速不稳/破音 → 讨论"为何不做逐字全流式" → 定为**句级流式**。随后又发现回复冗长/搜索拖慢延迟/风格死板，多轮迭代：

### 1) 句级流式（治首字延迟）
- 原实现：LLM 生成完整段回复 → 才整段送 TTS → 首字延迟 = STT + 全部LLM生成 + TTS首音（实测 17s+ LLM 独占）。
- 新实现 `_respond`：生产者(`producer`)边收 LLM token 边按句末标点（`。！？；!?…\n`，或超 40 字无标点则退而用逗号）切句入队；消费者逐句 `synthesize_stream` 合成+编码+推流，`carry` 跨句累积保证句间音频无缝。**首字只等第一句生成完，不等整段**。
- 打断：`asyncio.CancelledError` 网上抛，producer 一并 cancel；不影响原有 barge-in 机制。
- 回复气泡：仍在全部播完后补发一条 `sentence_start`(完整文本)，供设备显示气泡——音频早已流式播过，气泡只是事后补显示。

### 2) opus 编码器码率（治破音）
- `enc.bitrate = 24000`（16k 单声道语音场景默认档偏低，显式设更清晰）。

### 3) 慢工具垫话（治"搜索期间感觉卡住"）
- **关键讨论**：一开始想禁止泛知识问题触发 `web_search`（省搜索延迟），但用户指出"人口/GDP 这类数字是时效性的，禁搜索反而可能让模型说旧数据"——**该不该搜不该按问题类型定，该按答案是否含时效性事实定，这事交给模型自己判断更合理**（用户选项 B：完全放开，system prompt 不再限制搜索时机）。
- 真正要解决的是"搜索慢，用户没心理预期"，不是"别搜索"。→ `agent.py` 的 `tool_call` 事件里，若工具名在 `_SEARCH_TOOLS={web_search, datapro_search}`，一轮回复只在**首次**触发时插一句随机垫话（"我查一下"/"稍等，我查一下"/"我看一下啊"/"等我查查"）入 TTS 队列，播完接正式回答。垫话不计入气泡文字、不存历史。

### 4) 语音风格 prompt（`agent.py` `_system_prompt(profile, voice=True)`）
- 迭代了 3 版：① "最多2句/50字"硬字数限制 → 用户嫌死板；② 加"泛知识别搜索"限制 → 用户指出该按时效性判断非问题类型；③ **定稿**：只留"口语化、不用 markdown/书面符号、挑重点先说但别为凑短硬切"的方向性引导，字数和搜不搜都不再硬性规定，交给模型判断。

### 5) 顺手调的两个参数（用户直接指定）
- `_SILENCE_HANG`：8→17 帧（480ms→~1020ms），断句更宽容停顿。
- `_IDLE_TIMEOUT`：60→30s，聆听中无人说话回待命更快。

## 踩坑记录

- **`_strip_md`**：去 `** __ ~~ \`code\` #标题 *` 等 markdown 记号，避免气泡溢出 + TTS 把符号读出来。
- **气泡显示 vs 语音播放解耦**：气泡文字来自 `holder["reply"]`（累积所有 token，不含垫话），语音是句级实时流；两者数据来源不同但视觉/听觉上应保持一致（用户目前反馈良好，未发现不一致）。
